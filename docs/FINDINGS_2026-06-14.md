# Findings — 2026-06-14 (warm-started RL attempt → two real results)

Goal of the session: build a stronger agent via **warm-started self-play RL** (Tjong "SL first, then RL"

## 1. The "win8 = 0%" that drove the RL nulls was substantially a SCORING BUG

`score_terminals` (JAX RL reward) called the fan calculator as:

```python
fans = MahjongFanCalculator(..., verbose=False)
fan  = sum(fp * c for fp, c, *_ in fans)      # BUG
```

With `verbose=False` the calculator returns `(fanCount, fanName)` **2-tuples**, so the unpack bound
`c` to the **name string**; `fp * c` is string-repetition, and `sum(0 + str)` raises
`TypeError: unsupported operand type(s) for +: 'int'`. Caught by the surrounding `except` → **every win
scored fan = 0** → `win8` always reported 0% and `mean_reward ≈ 0`. The reward signal was *broken*,
not merely sparse.

Fix = `verbose=True` (returns `(fanValue, count, cn, en)`; total = `sum(fp*count)`). After the fix,
the warm-started SL policy in self-play:

| scorer | raw win | win8 (≥8 fan) | draw | mean reward |
|--------|--------:|--------------:|-----:|------------:|
| buggy (`verbose=False`) | 79.1% | **0.00%** | 21% | +0.00 |
| **fixed (`verbose=True`)** | 79.1% | **53.25%** | 21% | **+1.35** |

Implication: at least one "RL can't convert / win8=0" conclusion was an **instrumentation artifact**.
The prior from-scratch PPO run (`it3155 win8=0.00% rew=+0.00`) inherited the same broken scorer, so its
flat reward is not by itself evidence that RL failed to learn conversion. **The 16-null tally should be
re-audited against this scorer** before the bug is cited as evidence in the ToG paper.

### Production is NOT affected
The bug was confined to JAX training code (`train/jax_env/train_ppo_ws.py`, `train_ppo.py`,
`fan_reward.py:25`, `csm_validate.py`) that never shipped. The submitted/production bots handle the
calculator correctly:
- `bot/mahjong_bot.py:235`, `deploy/build/feature_agent.py:368`: `sum(cnt for cnt, _ in result)` —
  `verbose=False` 2-tuples, correctly summing the count. ✓
- deploy net-PIMC `deploy/caiest_cnn/csm_rollout.py:44`: `verbose=True`, correct 4-tuple unpack. ✓

So the uploaded `lad_chunjiandu` / net-PIMC entry's Hu-declaration and fan scoring are sound.

## 2. Warm-started RL with the full 40-block net is COMPUTATIONALLY INFEASIBLE (measured)

Validated infrastructure first: the JAX ResFused-40 forward matches the byte-exact numpy deploy net
(`numpy_resfused`) — argmax 16/16, max logit err 0.005. Obs = the deploy 38-plane encoding (`obs38`),
discard policy = `full_logits[:, 2:36]` (OFFSET_ACT['Play']=2), fresh value head on the 512-d
penultimate features. All correct.

But direct timing on an RTX A4000 (16 GB):

| stage | cost |
|-------|------|
| steady-state rollout (B=4096, N=90) | **558 s** (pure compute, not compile) |
| steady-state rollout (B=2048, N=55) | 171 s |
| PPO update | **5.3 s / minibatch × ~470 minibatches/iter ≈ 42 min** |
| frozen-trunk update | **still 5.3 s/minibatch** (each update still forward-passes 40 blocks) |

→ **~50 minutes per iteration** even with the trunk frozen; a useful run needs hundreds of iterations.
The 40-block forward dominates everything (≈1.9e15 MACs/rollout). This is the precise measurement behind
the project's standing "**forward-bound — RL won't be unlocked by compute**" finding.

**The only feasible warm-started RL is with a SMALL policy net**: distill 40-block → ~64ch×3-block
(~100× cheaper forward → seconds/iter), BC-warm-start it to the SL discard distribution, then self-play
RL. That is roadmap item R1; not attempted this session.
